# Reading Memory


With internal cheats,
reading values is relatively fast because there's no function call involved.
You simply read the data stored at an address (or address + offset).

With external cheats,
you'll be using ReadProcessMemory to grab the value. This can be slow.
To keep external cheats fast, you should minimise the amount of ReadProcessMemory calls you make.
Instead of reading 1 value, then 1 value, then 1 value,
read large blocks of data, like an entity struct, all at once.

Below are just some technques for reading externally.

## Techniques


### Have a fully padded struct?


e.g. your struct looks like this:
typedef struct _centity_t
{
char _pad1[0x02]; // 0x00
BYTE is_valid; // 0x02
char _pad2[0x19]; // 0x03
vec3 origin; // 0x1C
vec3ang view_angles; // 0x28
char _pad3[0x98]; // 0x34
int client_number; // 0xDD
enum _entity_type entity_type; // 0xD0
char _pad4[0xBC]; // 0xD4
enum _weapon_id weapon_id; // 0x190
char _pad5[0x2C]; // 0x194
int is_alive; // 0x1C0
} centity_t; // Size: 476 bytes


This is the best scenario.
You can read the entire struct by doing
Entity_t entity = { 0 };
ReadProcessMemory(h_process, entity_address, &entity, sizeof(Entity_t), NULL);


### Read @ struct address + member offset (most common)


REQUIRES HAVING A BIG LONG LIST OF STRUCT ADDRESS + MEMBER OFFSETS

This is the common method and the one I mentioned above that I think is slow.
unsigned int entity_addr = 0x734800;
unsigned int m_client_number = 0xDD;

int client_number = 0;
ReadProcessMemory(h_process, entity_addr + m_client_number, &client_number, sizeof(int), NULL);
