## Why do we need a file format for binaries?


Operating systems have 2 fundamental abstractions:
◇ files (binaries)
◇ processes

Binaries are a static representation of resources.
Processes are a dynamic representation of resources.

The process of transforming a static file into a dynamic process is called loading - done by a loader.

### The Loader


The loader is code that's part of the operating system.
In order to ‘load’ the program, the loader reads meta-information provided in the ELF file.
It reads information like
▪ the process's memory layout
▪ the address of the program's first instruction (the entry point)
▪ the addresses of each of the program's sections (.data, .text etc.)

All of this information is specified in the ELF header/file format.

The loader then uses this information to set the program up for execution by allocating memory for the program, creating data structures etc.

## The ELF Format


There are 4 main parts to the ELF format:

• ELF Header - gives a summary of the file - 32-bit/64-bit, endianness, entry point etc.
• ELF Section Header - describes the sections of the program and their location in the binary - .text .bss etc.
• ELF Program Header - defines the segments of the binary, which sections are in which segment, and the address of each of the segments when they're loaded into memory.
• ELF Symbol Table - stores information about variables, functions, objects, classes etc.
